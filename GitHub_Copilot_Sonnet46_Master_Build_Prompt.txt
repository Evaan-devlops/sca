======================================================================
A. PRODUCT OBJECTIVE
======================================================================

Create a generic, local-first SOP automation application for macOS.

A user should be able to enter a natural-language task in either GitHub Copilot
CLI or the direct Python CLI. The application must then:

1. Understand the requested business outcome.
2. Find and rank the relevant SOP or SOPs.
3. Select the relevant goal and ordered dependency closure.
4. Show the user the ordered steps required for the task.
5. Identify the required business skills.
6. Group demanded steps into canonical business capabilities.
7. Inspect the available tool catalogue.
8. Assign one mode to every capability:
   - NORMAL: an exact eligible ACTIVE + HEALTHY tool exists;
   - SETUP: no matching tool exists;
   - UPDATE: a matching tool exists but is stale, unhealthy, disabled,
     incompatible, or affected by a site change.
9. Set the current mode from the next runnable incomplete capability.
10. Execute the saved TaskPlan capability-by-capability.
11. Wait until the page is fully prepared before every browser action.
12. Pause safely for SSO, MFA, OTP, or manual authentication.
13. Keep the same live browser open while the user authenticates.
14. Resume only after the user types `continue` or `resume`.
15. Support SOP source files in both CSV and XLSX.
16. Generate a human-readable Markdown SOP and a validated execution manifest
    from either source format.
17. Support notes-first SOP rows by creating controlled agent-assisted proposals
    for missing fields.
18. Create or repair tools only for demanded capabilities that were actually
    executed and verified.
19. Persist exact progress, route evidence, corrections, build requests, and
    reports.
20. Produce a clean deterministic Mac release package.

This is a POC, but the core contracts must be real. Do not fake browser
liveness, readiness, tool activation, test success, or token usage.

D. SOP SOURCE CONTRACT: CSV AND XLSX
======================================================================

Support:

- `.csv`
- `.xlsx`

Do not support legacy `.xls`.

Canonical human-authored columns:

1. `step`
2. `step_id`
3. `action`
4. `element_name`
5. `element_type`
6. `value`
7. `expect`
8. `dependency`
9. `condition`
10. `condition_timing`
11. `if_true`
12. `if_false`
13. `notes`
14. `confirmation`

The active runtime must not use the legacy UI field `target`.

The word “target” may remain only where it is genuinely unrelated, such as a
branch target or project target directory.

Rules:

- `step_id` is unique and stable.
- `action` must use a validated enum.
- `element_name` is the visible or accessible business name.
- `element_type` is required for executable UI actions such as click, fill,
  enter, select, verify, and copy.
- `open` stores the URL in `value`, not `element_name`.
- `manual_auth` contains no password, OTP, token, or credential value.
- `expect` describes the observable postcondition.
- dependency and branch references must resolve to valid step IDs.
- incomplete notes-first rows may exist as NOTES_ONLY/DRAFT, but they must not
  execute until they become EXECUTABLE/CONFIRMED.

ElementType examples:

- page
- browser
- button
- link
- textbox
- textarea
- tab
- menu
- menu_item
- dropdown
- option
- checkbox
- radio
- row
- cell
- dialog
- status
- text

Action examples:

- open
- click
- fill
- enter
- select
- wait
- verify
- copy
- manual_auth
- end_success
- end_failure

CSV rules:

- UTF-8 and UTF-8 BOM
- clear malformed-row errors
- deterministic normalization

XLSX rules:

- prefer worksheet named `SOP`
- otherwise use the first visible worksheet
- first non-empty row is the header
- ignore fully empty trailing rows
- reject duplicate canonical headers
- reject formulas in operational cells
- reject empty/unreadable workbooks
- never execute macros or external links

Create one format-neutral loader:

```text
load_sop_source(path) -> normalized canonical rows
```

Both CSV and XLSX must pass through the same normalization and validation path.

Hashing:

- `source_file_sha256`: exact source bytes
- `source_content_sha256`: deterministic canonical row content
- `manifest_sha256`: deterministic manifest

Equivalent CSV and XLSX content must produce the same content hash and
equivalent Markdown step content.

Commands:
